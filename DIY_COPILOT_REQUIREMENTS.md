# DIY 版本：产品痛点与需求说明

本文档用于支撑在 Obsidian Copilot 代码库基础上自研一款符合我们工作流的产品。**不包含**对现有插件实现细节的复述，仅记录目标、痛点与可验收需求。

---

## 1. 目标

在保持「库内大模型辅助」核心能力的前提下，做一款**对话与会话可预期、可管理、上下文完全由人掌控**的助手产品，解决当前使用 Obsidian Copilot 时的混乱与重复会话问题。

---

## 2. 痛点（基于当前体验与预期落差）

- **从「历史」进入会话，状态不连续**  
  期望：每打开一个已存在的对话，**直接进入上次的完整对话状态**（同一会话、同一条目、不丢文）。  
  现状困扰：存在**被当成新对话**或**重复建会话**的情况，同一段交流在界面上或文件上出现两条「平行」会话，增加心智负担和清理成本。

- **对话与「对话列表」关系不清晰**  
  需要像聊天软件或 IDE 的会话一样：**一眼知道有哪些对话、哪条是当前、哪条是常用的**，而不是仅依赖散落的笔记文件或隐式行为。

- **Project 与单聊的关系若缺少一等公民设计**  
  工作往往按**项目/主题**切分；需要项目维度的**默认行为**（提示词、资料范围），且与**单条对话**的列表、打开方式一致、可发现。

- **新对话的上下文**  
  期望：**新建对话时绝不自动塞入**「当前活动笔记/网页等」为上下文，除非用户**显式选择**。避免每次新开一页就带上大块无关 context，也避免和「可预期列表」相冲突的隐式行为。

- **工作区与导航**  
  需要**可隐藏的侧边栏**承载对话列表，不占用主编辑区时仍可收起，需要时再展开。

---

## 3. 产品需求

### 3.1 会话与历史（核心）

- **R1. 单一事实来源**  
  任意时刻，「当前正在看的这条对话」在逻辑上**唯一**对应**一个**会话（含持久化键：如一个 vault 内笔记路径、或稳定 `sessionId`），**禁止**在「同一条目 / 同一会话 ID」上再静默新建并列会话导致重复。

- **R2. 从历史/列表进入即恢复**  
  从对话列表或等价入口**打开**某条会话时，应加载**该会话上次保存的完整状态**（至少包括消息历史；若我们定义更多状态如模型、已选 context，也一并恢复），**不**应无声切换成空白新会话。

- **R3. 显式「新建对话」**  
  仅当用户**明确执行「新建」**时，才创建一条**新**会话，并与已有条目区分（新标题 / 新 ID / 新文件）。

### 3.2 对话列表

- **R4. 清晰的对话列表**  
  提供**可浏览、可切换**的会话列表；每条有稳定标识（标题、时间、项目归属等，具体字段在实现时定）。

- **R5. Pin**  
  用户可将常用会话**置顶/固定**（pin），在列表中优先展示或独立分组（具体交互在实现时定）。

- **R6. 与 Project 的关联**  
  每个会话**应可归属**于某个 **Project**（或「无项目」），列表可按项目筛选或分组。

### 3.3 Project 能力

- **R7. 项目级默认系统提示**  
  在 Project 内可设置**默认提示词**；新建于该 project 下对话时，**默认**带该项目提示（用户仍可在该会话内覆盖，细节实现时定）。

- **R8. 项目级默认资料/上下文**  
  在 Project 内可设置**默认资料**（如文件夹、固定笔记集合、或与我们 schema 一致的资源列表）；**仅**在用户在该会话中未另行指定时，作为**可选项或默认值**使用——与 **R9** 一致，**不**在「仅新建空白会话」时未经确认就全量自动注入（具体「默认如何生效」在实现时写清：例如「进入会话时可选应用项目默认 / 或仅在用户点选应用」需二选一，避免与 R9 冲突）。

### 3.4 布局与侧栏

- **R9. 侧边栏对话列表**  
  提供**侧边栏**展示对话列表，支持**显示/隐藏**（折叠或关闭），不阻塞主工作区使用。

### 3.5 上下文的显式性（与 Copilot 差异点）

- **R10. 新建对话不自动加 context**  
  **创建新对话**或**清空起始状态**时，**不得**因「当前活动笔记/当前标签页/当前选区」等**自动**附加上下文。  
  所有进入模型上下文的**非系统、非项目默认策略内明确约定**的素材，须由用户**手动选择/确认**（如 @ 笔记、从项目默认资料中勾选等）。
